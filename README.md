## Запуск

```bash
# Запуск
mvn compile exec:java

# Тестирование
mvn test
```

Kotlin выбран как прагматичный JVM-язык, сокращающий шаблонный код по сравнению с Java при полной совместимости с ним. Ключевые возможности, использованные в проекте:

- **`data class`** - автогенерация `equals`, `hashCode`, `toString` и деструктуризации для `Book` без лишнего кода.
- **`sealed interface`** - исчерпывающая иерархия для `Command`. Компилятор гарантирует, что все варианты обработаны.
- **Функции высшего порядка** - `sortedBy`, `groupBy`, `minBy`, `maxBy` заменяют многословные цепочки `Comparator`.
- **Null-безопасность** - `toIntOrNull()` вместо блоков try/catch делает намерение валидации явным прямо в точке вызова.

### Архитектура

```
Main.kt           - цикл чтения и диспетчеризации (ввод -> имя команды + аргументы)
BookRepository    - владеет списком книг; маршрутизирует команды по имени
Command.kt        - sealed interface + один класс на команду
Book.kt           - data class; ID назначается счётчиком в companion object
```

Каждая команда - самодостаточная единица поведения. Добавление новой команды требует написать один класс, не трогая существующий код (принцип открытости/закрытости). `enum` смешал бы идентичность с поведением; цепочка `when` в диспетчере росла бы неограниченно и рассыпала логику.

**Почему `MutableList`, а не `Set` или конкурентная коллекция?**

В исходном скелете использовался `ConcurrentSkipListSet`, но это неудачный выбор:

1. `Book` потребовал бы реализации `Comparable` — у книги нет естественного порядка сравнения.
2. Удаление по ID всё равно требует O(n) сканирования списка, чтобы найти нужный элемент.
3. Это однопоточное CLI-приложение — защита от конкурентного доступа не нужна.

`MutableList` сохраняет порядок вставки (нужен для `LIST` без аргументов), поддерживает O(n) поиск по ID и не накладывает искусственных ограничений на доменную модель.

### Генерация ID

`Book.id` назначается счётчиком в companion object (`nextId++`), давая монотонно возрастающий уникальный идентификатор в рамках одной JVM-сессии. Счётчик не сбрасывается между тестами, поэтому тесты проверяют поведение (книга добавлена/удалена, вывод содержит ожидаемый текст), а не конкретные значения ID.
